# AlignMaster 输出模板

严格使用以下模板生成预审报告。

```markdown
# 📌 [需求名称] 产研测需求对齐预审报告

**审计负责人**：架构师和业务分析师 (AI 驱动)

---

## 1. 业务本质对齐 (Truth)

**一句话总结**：[谁] 在 [什么场景] 下，通过 [核心操作] 解决了 [什么问题]。

**核心链路图**：使用 Mermaid 语法绘制时序图或逻辑流程图。

---

## 2. 需求排雷清单 (Logic & Requirement Pits)

### 【理解偏差】
指出 PRD 中描述模糊、多处不一致的地方。

### 【异常缺失】
列出至少 3 个 PRD 未覆盖的异常分支（如：并发、断网、超时、重复提交、部分成功/部分失败、数据过期等）。

---

## 3. 代码与技术风险 (Technical Audit)

### 【代码实现冲突】
分析现有代码逻辑是否支持该需求，是否有硬编码障碍。若用户提供了代码/DDL/API，必须写出具体冲突点（类名、方法名、字段名、状态枚举等）。

### 【数据/性能隐患】
是否涉及存量数据清洗？是否存在慢查询或分布式一致性问题？是否有状态机死锁或循环依赖风险？

---

## 4. 评审会“灵魂拷问” (Alignment Questions)

### 问产品 (PO)
[针对逻辑漏洞、边界定义、异常流是否可接受的提问]

### 问研发 (RD)
[针对技术实现代价、兼容性、数据迁移与回滚的提问]

### 问测试 (QA)
[针对高风险边界场景、回归范围、验收标准的提问]
```

## 使用约束

- 如果输入不完整，也必须输出完整章节，并显式标注缺失信息。
- 若存在代码 context，第三部分必须包含“代码 vs 需求”的具体冲突点。
- 语气保持直接、专业、可执行，避免空泛描述。
